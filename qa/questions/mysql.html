特性聚簇索引（主键）二级索引
叶子节点完整数据行索引字段+主键值
查询直接拿数据可能需回表

覆盖索引：查询字段都在索引中，不需回表（Using index）。

非主键索引叶子节点存储索引字段+主键值，查到主键后需再查聚簇索引获取完整数据行，即回表。

减少回表：覆盖索引、联合索引包含查询字段。

  • count(*)：优先扫描最小的二级索引，MySQL 特殊优化
  • count(1)：等价于 count(*)
  • count(id)：需读取 id 字段判断 NULL，较慢

推荐用 count(*)。

InnoDB vs MyISAM 的 count：

  • MyISAM 维护了一个元数据计数器，count(*) 直接返回，O(1) 复杂度
  • InnoDB 没有维护计数器（因为 MVCC 导致不同事务看到行数不同），必须全表扫描或扫描索引
  • InnoDB 用 count(*) 会优先选最小的二级索引扫描，比扫描聚簇索引快

MVCC 实现原理：每行数据有隐藏列 DB_TRX_ID（最后修改事务ID）和 DB_ROLL_PTR（回滚指针，指向 undo log）。

  • ReadView：事务快照读时生成，包含当前活跃事务列表，用于判断版本可见性
  • RC 级别每次 SELECT 都生成新 ReadView；RR 级别只在第一次 SELECT 时生成，后续复用
  • 通过 DB_ROLL_PTR 沿版本链回溯，找到第一个对当前 ReadView 可见的版本

默认：可重复读（RR）

graph TD; A[幻读]-->B{读类型}; B-->|快照读|C["MVCC:复用ReadView"]; B-->|当前读|D[Next-Key Lock]; C-->C1[其他事务插入不可见]; D-->D1[Record+Gap Lock锁住范围]
  • 快照读：MVCC ReadView 复用，事务内只看开始时的快照
  • 当前读：Next-Key Lock（Record Lock + Gap Lock）锁住范围防止插入

MVCC + Next-Key Lock 解决大部分幻读，但复杂当前读场景仍可能出现。

  • 索引优化：添加合适索引、覆盖索引避免回表、联合索引遵循最左前缀
  • SQL 改写：避免 SELECT *、用 EXISTS 替代 IN、用 UNION 替代 OR
  • 表结构：垂直拆分（大字段分离）、水平分表（数据量大时）
  • 架构层：读写分离、加缓存（Redis）、ES 全文检索
  • 锁等待：用 SHOW PROCESSLIST 查看锁等待，information_schema 查锁信息

一、索引数据结构

类型结构特点适用场景
B+Tree 索引多叉平衡树，叶子节点双向链表InnoDB 默认，支持等值、范围、排序绝大多数查询场景
哈希索引哈希表O(1) 等值查询，不支持范围和排序Memory 引擎、自适应哈希
全文索引倒排索引支持分词搜索，MATCH AGAINST文本搜索（VARCHAR/TEXT）

二、索引逻辑分类

类型说明区别
聚簇索引主键索引，叶子节点存完整数据行每表只有一个，InnoDB 自动选主键
二级索引（辅助索引）非主键索引，叶子存索引字段+主键值可能需要回表查聚簇索引
联合索引多字段组合索引遵循最左前缀，字段顺序影响性能
覆盖索引查询字段全在索引中，不需回表EXPLAIN 显示 Using index，性能最优
前缀索引只索引字符串前 N 个字符节省空间，但无法用于 ORDER BY

三、场景对比

场景推荐索引原因
等值查询 =B+Tree / 哈希两者都支持，哈希更快但无范围
范围查询 > < BETWEENB+Tree叶子节点有序链表，天然支持范围
排序 ORDER BYB+Tree索引本身有序，避免 filesort
多条件组合联合索引一个索引覆盖多个条件，减少回表
文本模糊搜索全文索引分词匹配，比 LIKE '%keyword%' 高效

选型原则：优先 B+Tree（覆盖 90% 场景）；高频等值查询可考虑哈希；长文本字段用前缀索引节省空间；多条件查询用联合索引，区分度高的字段放前面。
