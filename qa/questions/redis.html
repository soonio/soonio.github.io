Redis 分布式锁实现：

  • 基础：SET key value NX EX 10
  • 问题1：锁过期但业务未完成 → 看门狗续期
  • 问题2：主节点宕机锁丢失 → Redlock（N个独立实例，超半数成功）
  • 释放锁：Lua 脚本保证原子性
if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("del",KEYS[1])
end

Fencing Token（防护令牌）：每次加锁时 Redis 返回单调递增的值（如版本号），资源方拒绝过期 token 的请求，可解决锁过期后的安全问题。

Redlock 争议：Martin Kleppa 指出 Redlock 在时钟跳跃场景下不安全。实际项目中，如果业务能容忍锁丢失（如幂等操作），用单节点 Redis + 看门狗足够；如果需要强一致锁，建议用 etcd。

方式原理特点
RDB（快照）某个时间点的全量快照恢复快，可能丢数据

RDB fork 与 COW：

  • RDB 通过 fork() 创建子进程，父子进程共享内存页（Copy-On-Write）
  • 只有被修改的内存页才会复制，大幅减少内存使用
  • 但如果 fork 时数据量大且写入频繁，COW 可能导致内存翻倍，甚至触发 OOM

AOF rewrite 过程：fork 子进程，将当前内存数据写为新 AOF 文件，同时父进程继续处理请求并追加到新 AOF 缓冲区，完成后替换旧文件。

方案特点
主从复制主写从读，异步复制，主宕机可能丢数据
哨兵（Sentinel）自动故障转移，无法水平扩展
Cluster16384哈希槽分片，支持水平扩展，多键用hashtag

Gossip 协议：Cluster 节点间通过 Gossip 协议传播状态信息（PING/PONG），包括节点上下线、槽位变更等。最终一致性，传播延迟与节点数相关。

扩缩容/resharding：通过 redis-cli --cluster reshard 迁移哈希槽到新节点，迁移过程中集群正常服务，支持在线扩缩容。
