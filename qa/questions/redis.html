etcd基于 Raft，强一致，不会丢锁

Redis 分布式锁实现：

  • 基础：SET key value NX EX 10
  • 问题1：锁过期但业务未完成 → 看门狗续期
  • 问题2：主节点宕机锁丢失 → Redlock（N个独立实例，超半数成功）
  • 释放锁：Lua 脚本保证原子性
if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("del",KEYS[1])
end
方案原理
Cache Aside先更新DB，再删除缓存
延迟双删先删缓存→更新DB→延迟再删缓存
异步删除（MQ）更新DB后发消息，消费者异步删缓存

其他一致性场景：

  • 内存缓存与 MySQL 不一致：快速过期策略（30秒）
  • 重复提交：Redis SET NX EX 分布式锁
  • 并发扣减：MySQL 悲观锁 SELECT FOR UPDATE

💼 实际案例

商品缓存采用 Cache Aside Pattern，更新商品信息时先更新DB再删除缓存。

String缓存、计数器、分布式锁

💼 实际案例

商品列表缓存用 String，SSE 早期集群方案用 Hash 管理连接，用户 Session 用 String。

方式原理特点
RDB（快照）某个时间点的全量快照恢复快，可能丢数据
AOF（追加日志）记录每个写命令数据安全，文件大恢复慢
混合持久化AOF重写时先写RDB格式再追加AOF兼顾两者优点
方案特点
主从复制主写从读，异步复制，主宕机可能丢数据
哨兵（Sentinel）自动故障转移，无法水平扩展
Cluster16384哈希槽分片，支持水平扩展，多键用hashtag
