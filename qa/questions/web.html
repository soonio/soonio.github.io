• 原理：二进制向量 + 多个哈希函数，快速判断元素是否存在
  • 返回"不存在"→ 一定不存在；返回"存在"→ 可能存在（有误判率）
  • 不能删除、不能遍历
  • 场景：缓存穿透拦截、黑名单过滤
算法原理特点
令牌桶固定速率放令牌，有令牌才能处理允许突发流量
漏桶固定速率流出平滑流量，不允许突发

go-zero 内置限流基于令牌桶。

补充：滑动窗口限流：将时间窗口分为 N 个小格，每个格独立计数，比固定窗口更平滑，避免窗口边界的突发流量。

分布式限流：基于 Redis + Lua 实现，用 INCR + EXPIRE 或滑动窗口算法，多个服务实例共享限流状态。

💼 实际案例

活动期间用 go-zero 内置限流保护核心接口。

服务调用失败率达到阈值时自动熔断，避免雪崩。

graph LR; A[Closed正常]-->|失败率超阈值|B[Open熔断]; B-->|冷却期结束|C[Half-Open试探]; C-->|成功|A; C-->|失败|B

go-zero 自适应熔断器：

  • 基于滑动窗口统计成功率和响应时间
  • 失败率超阈值自动触发熔断，快速失败
  • Half-Open 状态尝试放一个请求
  • 不需要手动配置阈值，自适应调整

补充：自适应熔断原理：go-zero 的 breaker 基于滑动窗口统计请求的成功率和平均响应时间，当 1.5×(请求数-成功数)/请求数 > 阈值 时触发熔断。与 Hystrix 的固定阈值不同，go-zero 动态调整，无需手动配置。

对比：Hystrix（Java，已停维）基于线程池隔离；Sentinel（Java）功能丰富但重；go-zero breaker 轻量、零配置，适合 Go 微服务。

结构：Header.Payload.Signature（三段式 Base64）

优点缺点
无状态、可跨域、不需要存 Session无法主动失效、Token 泄露风险

💼 实际案例

用户中心基于 JWT 实现 Token 管理，支持多端登录、Token 刷新、黑名单机制。

补充：Access + Refresh 双 Token 方案：

  • Access Token：短效（如 15 分钟），用于请求认证
  • Refresh Token：长效（如 7 天），用于刷新 Access Token，单独存储
  • Access Token 泄露风险小（有效期短），Refresh Token 可主动撤销

安全存储：前端用 httpOnly cookie 存储 Token（防 XSS），后端用 Redis 黑名单机制支持主动注销。
