业务只需要服务端单向推送（消息通知、活动推送），SSE 更轻量，自带重连机制，基于 HTTP 协议更易部署。WebSocket 适合双向通信场景，我们不需要。

  • 每个连接一个 goroutine，用 sync.Mutex + Map 管理连接
  • 服务端定时轮询 Map 发送心跳消息，检测连接状态
  • 客户端断开后通过 context 取消通知 goroutine 退出
  • 8核CPU占用仅20%，内存最多3.5G
  • 不限制并发连接数
  • 主要优化：提升 Redis 闲时连接数，提升连接创建速度
  • 调整 maxclients 和 tcp-keepalive 参数
  • 场景：活动期间，请求集中在商品列表、商品详情、预下单
  • 方案：内存缓存商品列表和商品详情，其他能缓存的接口也用内存缓存
  • 效果：成功支撑活动期间超2万QPS
  • 共享数据库 + tenant_id 字段隔离
  • GORM 中间件自动注入 tenant_id 条件
  • 联合索引 tenant_id 放最左边

go-zero 内置服务治理、限流熔断、服务发现等能力，开箱即用。相比自己封装，开发效率高，社区活跃。

K3S 更轻量，资源占用少，适合小规模部署。对于小布在家这种初创项目，K3S 足够满足需求，运维成本也更低。
