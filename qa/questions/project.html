业务只需要服务端单向推送（消息通知、活动推送），SSE 更轻量，自带重连机制，基于 HTTP 协议更易部署。WebSocket 适合双向通信场景，我们不需要。

补充：SSE 协议细节：

  • 基于 HTTP 长连接，Content-Type: text/event-stream
  • 自带 Last-Event-ID 支持断线重连，自动恢复未接收的消息
  • 数据格式简单（data: xxx\n\n），不需要额外协议解析

对比长轮询：长轮询每次连接超时后需重新建立，服务端无法主动推送；SSE 保持长连接，服务端可随时推送。

  • 每个连接一个 goroutine，用 sync.Mutex + Map 管理连接
  • 服务端定时轮询 Map 发送心跳消息，检测连接状态
  • 客户端断开后通过 context 取消通知 goroutine 退出
  • 8核CPU占用仅20%，内存最多3.5G

补充：连接管理细节：

  • 文件描述符：每个连接占用一个 fd，需调整系统 ulimit -n 和 K8S 资源限制
  • 内存计算：每个 goroutine 初始栈 2KB + 连接缓冲区，8 万连接约 3-4G
  • 优雅关闭：收到 SIGTERM 后停止接受新连接，等待现有连接处理完毕或超时强制关闭
  • 心跳机制：服务端定时发送心跳（如 30s），客户端超时未收到则重连
  • 不限制并发连接数
  • 主要优化：提升 Redis 闲时连接数，提升连接创建速度
  • 调整 maxclients 和 tcp-keepalive 参数
  • 场景：活动期间，请求集中在商品列表、商品详情、预下单
  • 方案：内存缓存商品列表和商品详情，其他能缓存的接口也用内存缓存
  • 效果：成功支撑活动期间超2万QPS

补充：完整高 QPS 方案：

  • 多级缓存：本地内存缓存（热点数据）→ Redis 缓存（温数据）→ MySQL（冷数据）
  • 负载均衡：Nginx/Ingress 轮询分发，多实例横向扩展
  • 连接池：数据库和 Redis 连接池复用，减少连接创建开销
  • 异步化：非核心链路异步处理（如日志、通知），减少响应时间
  • 降级策略：活动期间关闭非核心功能，保证核心链路稳定
  • 共享数据库 + tenant_id 字段隔离
  • GORM 中间件自动注入 tenant_id 条件
  • 联合索引 tenant_id 放最左边

补充：三种隔离模型对比：

模型隔离性成本
独立数据库最强最高（资源浪费）
共享库独立 Schema中等中等
共享库共享 Schema（我们）逻辑隔离最低

我们选择共享库+tenant_id，成本低、运维简单，配合 GORM 中间件保证数据隔离安全。

go-zero 内置服务治理、限流熔断、服务发现等能力，开箱即用。相比自己封装，开发效率高，社区活跃。

补充：与 Kratos 对比：

  • go-zero：内置服务治理、代码生成、限流熔断，开箱即用，适合快速开发
  • Kratos：更灵活的插件体系，gRPC 生态更好，适合大型团队定制
  • 选择 go-zero 主要因为团队规模小，开箱即用的能力更重要

K3S 更轻量，资源占用少，适合小规模部署。对于小布在家这种初创项目，K3S 足够满足需求，运维成本也更低。

补充：K3S 与 K8S 具体差异：

  • etcd 替换为 SQLite，减少资源占用
  • 移除了 alpha/beta 特性和云提供商插件
  • 单二进制文件，部署升级更简单
  • 内存占用约 512MB（K8S 需 2GB+）
  • 功能完全兼容 K8S API，迁移成本低
