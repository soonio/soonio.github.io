核心概念：

  • G（Goroutine）：用户态轻量级协程，初始栈仅 2KB
  • M（Machine）：内核级线程，与 OS 线程 1:1 映射
  • P（Processor）：逻辑处理器，数量由 GOMAXPROCS 决定，维护本地 G 队列

关键关系：M 必须绑定 P 才能执行 G。P 是 M 和 G 之间的桥梁。

graph LR; A[Goroutine创建]-->B{P本地队列}; B-->|未满|C[加入P本地队列]; B-->|已满|D[加入全局队列]; C-->E[M绑定P]; E-->F[M从P取G执行]; F-->G{G阻塞?}; G-->|网络IO或文件|H[M阻塞,P解绑找新M]; G-->|channel或Mutex|I[M+G阻塞,P解绑找新M]; G-->|执行完毕|J[M取下一个G]; J-->|本地队列空|K[Work Stealing]

调度策略：

  • 工作窃取：空闲 P 从繁忙 P 的队列中偷一半 G
  • 抢占式调度：Go 1.14+ 基于信号的异步抢占

G 阻塞时的行为：

阻塞类型M 行为P 行为
网络IO/文件交给 netpoller，M 休眠P 解绑，找新 M 继续执行
channel/MutexM+G 一起阻塞P 解绑，找新 M 继续执行

type SliceHeader struct {

💼 实际案例

8 万连接对象对 GC 有影响。优化：减少临时对象分配、使用 sync.Pool 复用。

  • 后进先出（LIFO）
  • 参数预计算：defer 参数在声明时确定
  • 与 return：先 return 赋值 → 再 defer → 最后 RET
func foo() (result int) {
    defer func() { result = 10 }()
    return 1 // 最终返回 10
}
  • 对象池，复用临时对象，减少 GC 压力
  • Get() 获取，Put() 归还
  • GC 时清空池中对象
  • 每个 P 有本地 pool，减少锁竞争
var pool = sync.Pool{
    New: func() interface{} { return new(bytes.Buffer) },
}
buf := pool.Get().(*bytes.Buffer)
buf.Reset()
defer pool.Put(buf)
