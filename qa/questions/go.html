核心概念：

  • G（Goroutine）：用户态轻量级协程，初始栈仅 2KB
  • M（Machine）：内核级线程，与 OS 线程 1:1 映射
  • P（Processor）：逻辑处理器，数量由 GOMAXPROCS 决定，维护本地 G 队列

关键关系：M 必须绑定 P 才能执行 G。P 是 M 和 G 之间的桥梁。

graph LR; A[Goroutine创建]-->B{P本地队列}; B-->|未满|C[加入P本地队列]; B-->|已满|D[加入全局队列]; C-->E[M绑定P]; E-->F[M从P取G执行]; F-->G{G阻塞?}; G-->|网络IO或文件|H[M阻塞,P解绑找新M]; G-->|channel或Mutex|I[M+G阻塞,P解绑找新M]; G-->|执行完毕|J[M取下一个G]; J-->|本地队列空|K[Work Stealing]

调度策略：

  • 工作窃取：空闲 P 从繁忙 P 的队列中偷一半 G
  • 抢占式调度：Go 1.14+ 基于信号的异步抢占

G 阻塞时的行为：

阻塞类型M 行为P 行为
网络IO/文件交给 netpoller，M 休眠P 解绑，找新 M 继续执行
channel/MutexM+G 一起阻塞P 解绑，找新 M 继续执行

💼 实际案例

SSE 服务 8 万并发连接，每个连接一个 goroutine，GMP 自动管理调度，8核CPU占用仅20%。

补充：sysmon 监控线程：Go runtime 有一个特殊的 sysmon 线程，不参与调度，以 10μs~500μs 的频率循环执行，职责包括：

  • 检测网络 IO 就绪（配合 netpoller）
  • 抢占长时间运行的 G（防止饿死）
  • 释放 M 持有的 P（M 阻塞超 10ms 时）
  • 触发 GC（每 2ms 检查一次是否需要 GC）

Goroutine 状态转换：

  • Gidle → Grunnable：新建 G 放入运行队列
  • Grunnable → Grunning：M 绑定 P 取出 G 执行
  • Grunning → Gsyscall：G 发起系统调用，P 解绑
  • Grunning → Gwaiting：G 阻塞在 channel/锁上
  • Grunning → Grunnable：G 执行完毕或主动让出（yield）

GOMAXPROCS 默认为 CPU 核心数，可通过环境变量或 runtime.GOMAXPROCS(n) 调整。它决定了同时能执行的用户级线程数。

SSE 服务用 sync.Mutex + Map 管理连接池。早期用 sync.Map（集群方案），后换成 Mutex+Map（单机方案），性能更好。

补充：RWMutex 读写锁：读多写少场景下，sync.RWMutex 比 Mutex 性能更好，允许多个读锁并发持有，但写锁独占。使用 RLock()/RUnlock() 管理读锁。

竞态检测：Go 内置 race detector，测试时加 -race 标志（go test -race），可自动发现数据竞争问题，是 CI 最佳实践。

死锁检测：Go runtime 能检测部分死锁（所有 goroutine 都在等待时 panic）。复杂场景可用 pprof 查看 goroutine 阻塞堆栈定位问题。

方式一：context.WithTimeout（推荐）

ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
select {
case result := <-doSomething(ctx):
    // 正常处理
case <-ctx.Done():
    return ctx.Err()
}

方式二：select + time.After

select {
case result := <-ch:
case <-time.After(3 * time.Second):
    // 超时
}

context 可沿调用链传递，子 goroutine 也能感知超时；time.After 简单但无法传递上下文。

type SliceHeader struct {
    Data uintptr // 指向底层数组
    Len  int     // 当前长度
    Cap  int     // 容量
}
  • slice 共享底层数组，修改一个可能影响另一个
  • append 时 cap 不够会分配新数组，原 slice 不变
  • 扩容规则：cap<256 → 2倍；cap>=256 → 1.25倍+192

nil slice vs empty slice：

  • var s []int（nil slice）：Data 为 nil，len=0，cap=0
  • make([]int, 0)（empty slice）：Data 指向 zerobase，len=0，cap=0
  • JSON 序列化差异：nil slice → null，empty slice → []

子切片内存泄漏：从大 slice 中取子切片（s[1000:1001]），子切片仍引用整个底层数组，导致大数组无法被 GC 回收。解决：用 copy 创建独立副本。

// 避免内存泄漏的写法
sub := make([]byte, 1)
copy(sub, big[1000:1001])
// 此时 big 可被 GC 回收

底层是哈希表（bucket 数组 + 溢出桶）：

  • 每个 bucket 最多存 8 个键值对
  • 哈希冲突：链地址法，溢出桶链表扩展
  • 负载因子 > 6.5 时渐进式扩容
  • 并发读写会 panic

并发安全方案：sync.Mutex+map（通用）、sync.Map（读多写少）、sync.RWMutex+map（读写分离）

💼 实际案例

SSE 连接管理用 map，key 是连接 ID，用 sync.Mutex 保证并发安全。

补充：渐进式扩容：map 扩容不是一次性迁移所有数据，而是每次插入时逐步迁移旧 bucket 的内容，避免单次扩容造成性能抖动。

迭代顺序随机化：Go 的 map 迭代顺序是随机的（从 1.0 开始），即使同一个 map 两次遍历，元素顺序也不同。这是为了防止代码依赖特定迭代顺序。

nil interface 陷阱：一个常见 bug：

func do() error {
    var p *MyError = nil
    return p // 返回非 nil 的 error！
}
err := do()
fmt.Println(err != nil) // true！因为 type 不为 nil

interface 内部存储了类型信息，即使 value 为 nil，interface 本身也不等于 nil。这会导致 if err != nil 判断出现意外结果。

动态派发开销：interface 方法调用需要动态派发（查 itable），比直接调用多一层间接寻址。性能敏感场景应使用具体类型。

8 万连接对象对 GC 有影响。优化：减少临时对象分配、使用 sync.Pool 复用。

补充：GOGC 调优：

  • GOGC 控制 GC 触发频率，默认 100（堆增长到 2 倍时触发 GC）
  • 可通过 GOGC=50 环境变量调整，降低值更频繁回收但增加 CPU 开销
  • Go 1.19+ 引入 GOMEMLIMIT，设置软性内存上限，超过时 GC 更积极运行
  • 容器环境中 GOMEMLIMIT 特别有用，避免 OOM 被 kill

GC Pacer：GC 开始时启动 Pacer，控制标记节奏，确保在堆分配完之前完成标记，同时最小化 STW。

内存分配器：Go 采用线程缓存分配器（TCMalloc 思想）：

组件作用
mspan连续内存页，按大小分类
mcentral所有 P 共享，缓存 mspan

  • 后进先出（LIFO）
  • 参数预计算：defer 参数在声明时确定
  • 与 return：先 return 赋值 → 再 defer → 最后 RET
func foo() (result int) {
    defer func() { result = 10 }()
    return 1 // 最终返回 10
}

panic recover：defer + recover 是 Go 处理 panic 的唯一方式：

func safe() {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("recovered: %v", r)
        }
    }()
    panic("something wrong")
}

注意：recover 只在 defer 函数内调用才有效。defer 的 LIFO 特性确保资源按正确顺序释放。

性能影响：Go 1.14 后 defer 性能大幅优化（延迟插入），开销约 25ns，不必过度担心。但在极高频循环中仍需注意。

  • 对象池，复用临时对象，减少 GC 压力
  • Get() 获取，Put() 归还
  • GC 时清空池中对象
  • 每个 P 有本地 pool，减少锁竞争
var pool = sync.Pool{
    New: func() interface{} { return new(bytes.Buffer) },
}
buf := pool.Get().(*bytes.Buffer)
buf.Reset()
defer pool.Put(buf)

💼 实际案例

可用于 SSE 消息对象复用，减少 GC 压力。

补充：内部实现：

  • 每个 P 维护本地 pool 缓存（poolLocal），Get 优先从本地取，最小化锁竞争
  • 本地为空时从其他 P 偷取（work stealing 思想）或调用 New 函数创建
  • Put 时优先放回本地 pool，本地满则放入共享的 poolShared

注意：sync.Pool 中的对象可能在 GC 时被清空，所以 Get 后必须检查/重置状态（如 buf.Reset()）。

适用场景：大量短生命周期的临时对象复用，如 bytes.Buffer、encoding/json 编解码器等。
