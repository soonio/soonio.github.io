• 基于 go-zero 内置的 etcd 实现服务注册与发现
  • etcd 集群自行搭建，3 个节点 + 1 个代理节点，保证高可用
  • 不是所有服务都部署在 K8S 上，所以用 etcd 而不是 K8S Service
  • 每个服务保持可集群扩展的能力，随时可以增加副本

补充：服务发现细节：

  • 健康检查：服务实例定期发送心跳（TTL 机制），etcd 自动剔除不健康实例
  • Watch 机制：客户端 Watch etcd 的 key 变化，实时感知服务上下线
  • 本地缓存：客户端缓存服务列表，避免每次调用都查询 etcd，降低延迟

主要用 go-zero 内置能力：

  • 负载均衡：基于 etcd 服务发现自动分配请求到不同实例
  • 熔断：内置自适应熔断器（详见 Q29）
  • 限流：内置令牌桶限流（详见 Q28）
  • 超时控制：服务间调用设置超时时间，防止级联故障
  • 重试：调用失败自动重试，提高可用性

补充：重试风暴：重试可能导致请求放大，需配合指数退避 + 抖动策略，避免雪崩。go-zero 内置自适应重试。

灰度发布：基于请求头或用户 ID 路由到灰度实例，逐步放量，降低发布风险。

部署架构：

  • K8S 同集群多实例部署，部分服务不在 K8S 上
  • 配置管理：基于 K8S ConfigMap
  • 监控链路：OpenTelemetry 链路追踪 + Prometheus + Grafana 监控告警
  • 日志系统：Promtail 采集 → Loki 存储 → Grafana 查询

集群部署问题：

  • 服务间通信延迟：通过服务发现直接调用，减少中间环节
  • 配置同步：用 ConfigMap 统一管理，修改后重新部署生效
  • 实例扩缩容：基于 etcd 自动注册新实例

补充：CI/CD 流程：

  1. 代码提交 → GitLab CI 触发构建
  2. Docker 镜像构建 → 推送镜像仓库
  3. K8S 滚动更新（Rolling Update），逐步替换旧实例
  4. 健康检查通过后继续下一批，失败自动回滚

部署策略：滚动更新（零停机）为主，重要服务用蓝绿部署（双版本并行，一键切换）。

  • 基于 Redis 实现无状态 Token
  • 用户登录后生成 Token，存储到 Redis 中，包含用户信息和权限
  • 多个服务通过 Redis 共享认证状态，不需要 Session 复制
  • Token 过期后自动失效，支持主动注销（删除 Redis 中的 Token）

补充：Token 刷新机制：Access Token 短效（15min），Refresh Token 长效（7天），Access 过期后用 Refresh 获取新 Token。

多端登录：每个设备生成独立 Token，存储到 Redis Hash 中，支持查看在线设备列表并强制下线。

  • 业务层保证幂等性，重复消费不会产生副作用

问题：单条写入 Redis Stream 产生巨大写入压力

方案：基于 Go channel 实现批处理器

  • 业务消息先写入 Go channel（内存缓冲区）
  • 数量阈值：缓冲区消息达到 100 条触发
  • 时间阈值：超过 100ms 即使没达到数量也触发
  • 使用 Redis pipeline 批量写入

本质：生产者-消费者模型，Go channel 作为中间缓冲区，削峰填谷。

  • Redis 5.0 内置数据结构，不需要额外部署消息队列服务
  • 支持消费者组、消息确认、历史消息回溯
  • 对于 SSE 场景，消息量不是特别巨大，Redis Stream 足够用
  • 如果消息量特别大，可以用 RocketMQ 或 Kafka 替代
  • 优点：架构简单、运维成本低、延迟低
  1. 消息生产：业务服务产生消息，写入消息队列
  2. 消息存储：消息队列持久化存储，支持重试和回溯
  3. 消息消费：推送服务消费消息队列，根据用户连接关系分发
  4. 连接管理：每个用户连接对应一个 goroutine，Map 管理映射关系
  5. 推送协议：SSE（单向推送）或 WebSocket（双向通信）
  6. 高可用：多实例部署，消息队列保证消息不丢失
  7. 性能优化：批量写入、内存缓存、连接数限制和流控
